# PyTorch XPU Skip Decorator Management

## Find skip markers

Search the test file and OpInfo definitions:

```bash
# Method-level decorators
grep -n "skipXPU\|skipIfXpu\|xfailIfXPU\|expectedFailureXPU" <test_file>

# Inline conditional skips
grep -n "device_type='xpu'" <test_file>

# OpInfo DecorateInfo entries
grep -n -A2 "DecorateInfo.*skip.*xpu\|DecorateInfo.*xpu.*skip" \
  torch/testing/_internal/common_methods_invocations.py

# instantiate_device_type_tests dict entries
grep -n "skip_xpu\|xfail_xpu" <test_file>
```

Patterns to recognize:

| Pattern | Location |
|---|---|
| `@skipIfXpu` | Decorator on test method |
| `@unittest.skipIf(..., "XPU ...")` | Decorator on test method |
| `@expectedFailureXPU` | Decorator on test method |
| `@xfailIfXPU` | Decorator on test method |
| `DecorateInfo(unittest.skip("..."), device_type='xpu')` | Inside `OpInfo` `decorators` list |
| `skip_xpu`, `xfail_xpu` dict entries | Passed to `instantiate_device_type_tests` |
| `skipIfXpu` in conditional blocks | Inline skip logic in test body |

## Remove a skip marker

1. Delete the decorator line (or the `DecorateInfo` entry from the list).
2. Clean up unused imports if this was the last usage of that decorator.
3. For `OpInfo` `DecorateInfo` entries: remove the entry from the `decorators`
   list; if it was the only entry, remove the `decorators` kwarg entirely.

**Dynamic test names:** many test classes are generated by
`instantiate_device_type_tests` (e.g. `TestCommonXPU` from `TestCommon`).
If a grep on the class name fails, search for the base class:

```bash
grep -n "class TestCommon[^X]" <test_file>
grep -n "instantiate_device_type_tests" <test_file>
```

## Temporarily remove for reproduction

When `fix/reproduce` finds all tests skipped by `@skipIfXpu`:

1. Remove the decorator.
2. Re-run the test to confirm the underlying failure.
3. Revert: `git checkout <test_file>`.

The skip is often masking the actual bug — never return `CANNOT_VERIFY` solely
because all tests are skipped.

## Add a new skip (only when allow_skip=true)

Only when the orchestrator explicitly sets `allow_skip=true` AND the fix
genuinely requires significant implementation work (missing kernel, blocked
upstream feature, dependencies on onednn/triton etc).

**Requirements before adding:**
1. A tracking issue must exist. File one if it does not.
2. Record the skip in the orchestrator's summary report.

Format:

```python
@skipIfXpu  # TODO: <short reason>. Tracking: <repo>#<issue_number>
def test_something(self):
    ...
```

This is a temporary measure. The tracking issue owns the real fix.

## HARD RULES
- NEVER add a skip without a tracking issue number.
- NEVER add a skip when `allow_skip=false`.
- NEVER treat an existing skip as proof the issue is already fixed.
