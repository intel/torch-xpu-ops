# EU Stall Attribution

Attribute high EU stall for one XPU target kernel by combining stall reason counters, per-IP stall data, occupancy, ASM extraction, and ASM/source mapping.

This skill starts after a target kernel is known. It complements `kernel-perf-analysis`: roofline/counter closure can show where a memory-hierarchy or compute gap exists, but this skill explains why EU stall happens at the instruction/source level.

## When to Use

Use this skill when:
- `XVE_STALL` is a dominant deficit.
- `eu-utilization-triage` routes to stall attribution.
- Roofline/counter comparison identifies a suspicious gap but not the stalled instruction path.
- You need a PR-ready attribution card with stall class, stage, hot IPs, source lines, and next lever.
- You need to distinguish Dist/Pipe/Send/Sbid/PSDep/Control/InstrFetch/Sync stall reasons and map hot stall IPs back to source.

Do not use this skill when stall is low and the main issue is low co-issue; use `eu-ilp-coissue`. Do not use it for broad workload analysis.

## Inputs

| Input | Meaning |
|---|---|
| `repro-cmd` | Command that launches the target kernel |
| `kernel-filter` | Target kernel filter |
| Baseline counters | `ComputeBasic` three-state and occupancy |
| Stall counters | `VectorEngineStalls` or stall-sampling output when available |
| ASM artifacts | Generated by `extract-xpu-kernel-asm`, required for L4 evidence |

## Workflow

### Step 1: Collect Stall Evidence

Collect as much of the following as the platform supports:

| Evidence | Purpose |
|---|---|
| `ComputeBasic` | Baseline `XVE_STALL`, active/co-issue split, occupancy |
| `VectorEngineStalls` | Class-level stall ranking; use for ranking, not additive totals |
| Stall sampling | Per-IP absolute stall counts; preferred source for hot instruction attribution |

Treat stall sampling per-IP absolute counts as the primary evidence when available. Use `VectorEngineStalls` for class ranking only because stall classes can overlap and should not be added together.

If `VectorEngineStalls` and stall sampling are unavailable, run a controlled `ComputeBasic` sweep with lower vs higher resident-thread configurations. If `XVE_STALL` drops with more resident threads, classify the stall as likely Stage-1 candidate starvation; if it stays high, classify it as likely Stage-2 resource contention. Mark this fallback as lower confidence than per-IP attribution.

### Step 2: Determine Dominant Stall Class and Stage

Classify the dominant stall class from per-IP counts when possible.

| Stage | Typical classes | Meaning |
|---|---|---|
| Stage-1 candidate starvation | Dist, Sbid, PSDep, Control, InstrFetch, Sync | The scheduler lacks ready candidates |
| Stage-2 resource contention | Pipe, Send | Candidates exist but a resource is busy |

Use occupancy to refine routing:
- Stage-1 with low/moderate occupancy often routes to `eu-tlp-occupancy`.
- Stage-1 with high occupancy and dependency distance issues often routes to `eu-ilp-coissue`.
- Stage-2 Pipe routes to instruction-shape or pipe-balance work.
- Stage-2 Send routes to memory/locality/coalescing work.

### Step 4: Extract ASM

Use `extract-xpu-kernel-asm` for the target kernel. Preserve:
- ASM directory.
- Kernel name and signature.
- Any debug line information.
- Compile path type, such as SYCL JIT, SYCL AOT, Triton, or oneDNN ngen.

### Step 5: Map Hot IPs to Source

Use `asm-source-mapping` with:
- ASM directory.
- Top hot IPs from stall sampling.
- Kernel filter.

Output an IP/source table:

| IP | Stall class | Count | ASM line | Source file:line | Construct |
|---|---|---:|---:|---|---|

Mapping priority:
1. Debug line / DWARF when available.
2. IGC comments or embedded line info.
3. Kernel signature and structural source matching.
4. Pattern recognition fallback, explicitly marked as lower confidence.

## Routing Rules

| Attribution result | Next action |
|---|---|
| Stage-1 dependency distance, high occupancy | `eu-ilp-coissue` |
| Stage-1 candidate starvation, low/moderate occupancy | `eu-tlp-occupancy` |
| Stage-2 Pipe | instruction-shape fix or pipe balancing |
| Stage-2 Send | memory hierarchy / locality / coalescing optimization |
| Sync or Control dominant | algorithm or synchronization restructure |
| InstrFetch dominant | kernel size or instruction-cache investigation |

## Pitfalls

- Do not add VES stall classes together; classes can overlap.
- Do not claim source-level root cause without ASM/source mapping.
- Do not route Stage-2 Pipe/Send to TLP blindly; more threads can worsen resource contention.
- Do not rerun downstream skills' collection steps when this skill already produced the handoff evidence.
